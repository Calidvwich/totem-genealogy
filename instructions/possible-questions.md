# 验收可能提问与简要回答

本文用于准备项目验收答辩。问题按前端、数据库、数据库逻辑、实际业务场景四部分整理，回答以“能快速说明设计理由和实现方式”为主。

---

## 一、前端部分

### 1. 前端使用了什么技术？

前端是单页页面，主要由 `interface.py` 生成 HTML、CSS 和 JavaScript。页面通过 `fetch` 调用 FastAPI 后端接口，图表使用 ECharts，族谱树也使用 ECharts 的 tree/graph 展示能力。

### 2. 为什么没有使用 Vue 或 React？

项目重点是数据库系统实践，前端主要服务于业务演示和数据库操作验证。使用原生 HTML、CSS、JavaScript 可以减少依赖，方便在 Ubuntu 18.04 和 Python 3.6 环境下运行。

### 3. 登录状态如何维护？

当前采用轻量前端状态维护。登录成功后，前端把当前用户信息保存在页面的 `state.user` 中，后续需要权限的请求会携带 `current_user_id`。同时后端会返回 `log_session`，用于记录本次用户操作日志。

### 4. 为什么需要登出功能？

登出可以清空前端当前用户状态，隐藏管理页面，并记录 `log-out` 操作日志。这样可以避免多人使用同一浏览器时权限状态混乱。

### 5. 页面如何区分管理员和普通用户？

登录后根据 `user_id` 判断是否为 `admin`。管理员可以看到用户管理入口，普通用户只能注册、登录、管理自己有权限的族谱和成员。

### 6. 普通用户能做什么？

普通用户可以登录、注册、创建族谱、编辑自己创建或被授权的族谱成员，也可以查看族谱、查询成员和浏览树形图。

### 7. 管理员能做什么？

管理员可以查看用户列表、新增用户、修改用户、删除用户，也可以执行生成数据导入、整库导出等高权限操作。

### 8. 前端如何处理接口报错？

统一通过 `api()` 函数封装 `fetch`。如果响应不是 2xx，会解析后端返回的 `detail` 字段，并把 Pydantic 参数错误转换成可读提示，避免出现 `[object Object]`。

### 9. 为什么搜索结果需要显示耗时和内存？

这是第八部分索引与性能实验的要求。搜索接口会返回是否成功、耗时、内存占用、结果数量和当前模式，用于比较普通模式和性能模式。

### 10. 普通模式和性能模式在前端如何切换？

页面上有性能模式开关。关闭时执行普通查询并尝试关闭索引；开启时创建相关索引后执行查询，并显示索引启用状态和性能指标。

### 11. 族谱树是如何展示的？

前端调用后端 `/api/tree` 获取树数据，再用 ECharts 动态渲染。节点支持缩放、重置视图、点击姓名查看详情、点击节点圆点展开或隐藏关系树。

### 12. 为什么男性节点是蓝色，女性节点是粉色？

这是为了增强可视化识别能力。性别是族谱系统中的核心属性，颜色区分可以帮助快速理解成员结构。

### 13. 点击姓名和点击节点圆点有什么区别？

点击姓名框只展示成员详情；点击圆点只负责展开或隐藏该成员的关系树。这样避免“想看详情却展开树”或“想展开树却打开详情”的操作冲突。

### 14. 成员详情页面展示哪些信息？

展示成员基础信息、所属族谱、父母、配偶、子女、照片、婚姻关系等。照片优先展示数据库中最后上传的图片，否则显示默认头像 `resources/defaultpic.jpg`。

### 15. 前端如何上传照片？

用户在成员编辑或详情相关操作中上传本地图片，前端使用 `FormData` 传给后端。后端计算 sha256 后把图片内容存入数据库，不保存为本地图片文件。

### 16. 为什么页面需要导入和导出按钮？

导入导出是数据库系统常见维护功能。导入支持 CSV 新建族谱、生成数据导入、导出包恢复；导出支持整库、多个族谱、单个成员信息。

### 17. 导出族谱为什么支持多选？

现实中可能只需要迁移或备份部分族谱。多选可以一次导出多个相关族谱，避免只能导出单个族谱造成操作重复。

### 18. 为什么前端查询成员时遇到重名要填写 ID？

姓名不是唯一键，族谱中重名很常见。为了避免误操作，系统在出现多个精确匹配姓名时要求用户输入具体成员 ID。

### 19. 统计查询为什么也要支持姓名和 ID？

老师或用户通常更习惯输入姓名；但数据库中主键是 ID。系统先支持姓名查询，遇到重名再要求 ID，兼顾易用性和准确性。

### 20. 为什么首页性别比使用扇形图？

性别比是概览指标，用扇形图可以直观看出男性、女性、未知性别占比。全库浏览时展示全库比例，进入具体族谱时展示该族谱比例。

## 二、数据库部分

### 1. 使用的数据库是什么？

使用 TotemDB，默认数据库名是 `genealogy`，用户是 `totem`，端口使用 `55432`，避免和默认 `5432` 端口上的其他数据库冲突。

### 2. 为什么使用 55432 端口？

本机或 WSL 中可能已有 PostgreSQL/TotemDB 服务占用 `5432`。使用 `55432` 可以降低端口冲突概率，启动脚本也默认使用这个端口。

### 3. 项目有哪些核心表？

核心表包括：

- `users`：系统用户。
- `genealogies`：族谱。
- `collaborations`：族谱协作权限。
- `members`：族谱成员。
- `marriages`：婚姻关系。
- `member_photos`：成员照片。

### 4. 用户表主要字段是什么？

`users` 包括内部自增 ID、登录账号 `user_id`、密码哈希 `password_hash`、用户名 `username` 和创建时间。

### 5. 族谱表主要字段是什么？

`genealogies` 包括族谱 ID、标题、姓氏、修订时间和创建者 ID。创建者默认是当前登录且创建族谱的用户。

### 6. 成员表主要字段是什么？

`members` 包括成员 ID、族谱 ID、姓名、性别、出生年、死亡年、父亲 ID、母亲 ID、世代、简介和头像 sha256。

### 7. 婚姻表主要字段是什么？

`marriages` 包括婚姻 ID、族谱 ID、配偶 A、配偶 B、结婚年份和离婚年份。

### 8. 照片为什么单独建表？

成员表只保存照片 sha256，图片内容放在 `member_photos` 中，可以避免成员表过大，也方便多个成员复用同一个图片哈希。

### 9. 照片为什么用 sha256 存储？

sha256 可以作为图片内容的唯一指纹，避免重复存储相同图片，也能保证成员表只引用图片摘要。

### 10. 默认头像如何处理？

如果成员没有上传照片，后端返回本地默认图片 `resources/defaultpic.jpg`。只有用户上传过照片时才从数据库读取图片内容。

### 11. 密码如何存储？

密码使用 PBKDF2-SHA256 哈希存储，包含算法、迭代次数、盐和哈希值。数据库中不保存明文密码。

### 12. 为什么要迁移明文密码？

早期测试中可能出现明文密码。系统登录时可以识别旧格式并迁移为哈希，另有迁移脚本用于批量修复。

### 13. 数据库有没有外键？

为了兼容当前 TotemDB 对部分高级 SQL 的支持限制，表结构没有强依赖外键约束，跨表一致性主要由应用层校验和导入脚本保证。

### 14. 没有外键会有什么风险？

风险是可能出现孤立成员、无效父母 ID、无效婚姻关系等。项目通过新增/修改时校验、导入时校验、数据生成约束和检查脚本降低风险。

### 15. 成员 ID 是如何生成的？

真实数据库模式下创建成员时查询当前最大 `member_id`，取 `MAX(member_id)+1`。批量生成数据则由生成脚本预先分配 ID。

### 16. 族谱 ID 是如何生成的？

创建族谱时同样会基于当前最大族谱 ID 生成新的 ID，批量导入时使用生成文件中的族谱 ID。

### 17. 当前数据规模是多少？

当前完整导出大约包含 105004 个成员、16183 条婚姻关系、11 个族谱、3 个用户和 12 条协作权限。

### 18. 完整数据库导出在哪里？

完整导出在 `fulldb/` 目录，核心文件是 `fulldb/import_bundle.json`。可以用 `import.py restore fulldb/import_bundle.json --reset` 恢复。

### 19. 导出的 CSV 和 import_bundle 有什么区别？

CSV 便于人工查看和审计；`import_bundle.json` 是可直接导入恢复的结构化导出包，包含 manifest 和各表数据。

### 20. 导入时如果重复怎么办？

导入包恢复时会检查用户账号、主键 ID、照片 sha256、协作关系等是否重复。发现重复会停止导入并输出原因。

### 21. 为什么导入时默认会停止而不是覆盖？

覆盖可能破坏已有数据。停止导入可以避免误覆盖，只有使用 `--reset` 时才明确表示要完整替换数据库内容。

### 22. 如何初始化空数据库？

先创建 `genealogy` 数据库，再执行 `init_db.sql` 创建表结构。之后可以选择导入生成数据或恢复 `fulldb/import_bundle.json`。

### 23. 数据库连接如何实现？

后端通过 `TotemClient` 调用 TotemDB 的 `tsql` 命令执行 SQL。环境变量控制数据库名、用户、端口和 `tsql` 路径。

### 24. 为什么没有直接使用 Python 数据库驱动？

项目环境中主要依赖 TotemDB 自带命令行工具。为了兼容课程环境，使用 `tsql` 作为数据库访问入口。

### 25. SQL 注入如何防护？

项目中对字符串使用 `sql_literal()` 做单引号转义，路径参数多数由 FastAPI/Pydantic 转为整数。虽然不是驱动级参数化，但避免了直接拼接原始字符串。

### 26. 数据库索引有哪些？

性能模式会创建成员表相关索引，例如族谱 ID、姓名、父亲 ID、母亲 ID 等字段索引，用于加速成员搜索和亲属查询。

### 27. 普通模式和性能模式对数据库有什么影响？

普通模式会尝试关闭性能索引；性能模式会创建索引。两种模式都执行同类搜索，并返回耗时和内存信息，便于对比。

### 28. EXPLAIN 输出在哪里？

性能实验中的 `EXPLAIN ANALYZE` 输出保存在 `output/performance-test/`，文件名按日期和时间生成。

### 29. 日志是否写入数据库？

当前日志不写入数据库，而是写到本地 `log/` 目录。错误日志在 `log/errorlog.txt`，用户操作日志在 `log/userlog/`。

### 30. 为什么日志不放数据库？

日志主要用于本地验收和调试，不属于核心业务数据。写文件实现简单，也避免日志写入失败影响数据库业务操作。

## 三、数据库逻辑部分

### 1. 父母关系如何表示？

成员表中用 `father_id` 和 `mother_id` 表示父母。它们引用同一张成员表中的成员 ID。

### 2. 为什么父母关系放在成员表里？

父母关系是一对多关系，一个孩子最多有一个父亲和一个母亲，直接放在成员表里查询简单，也便于构造树。

### 3. 婚姻关系为什么单独建表？

婚姻是成员之间的多对多历史关系，且有结婚年份和离婚年份等属性，适合单独建 `marriages` 表。

### 4. 父母和孩子关系如何自动生成婚姻？

当新增或修改成员时，如果同时存在父亲和母亲，系统会检查二者是否已有婚姻关系；如果没有且满足校验，就自动创建婚姻关系。

### 5. 删除孩子关系为什么会影响婚姻？

如果某对父母的婚姻关系是由共同子女自动生成的，当移除最后一个共同子女关系后，这条自动婚姻关系就没有业务依据，可以被移除。

### 6. 离婚后孩子关系是否删除？

不会。孩子已经出生后，即使父母离婚，父子/母子关系仍然存在，系统不会因为离婚删除亲子关系。

### 7. 父亲和母亲性别如何校验？

父亲必须是 `M`，母亲必须是 `F`。如果用户输入的父母姓名匹配到错误性别，系统会拒绝保存。

### 8. 父母和孩子是否必须属于同一族谱？

必须属于同一族谱。跨族谱设置父母会造成族谱树混乱，因此新增和修改成员时会校验族谱 ID。

### 9. 年份约束有哪些？

主要约束是父母出生年份 < 结婚年份 < 孩子出生年份 < 离婚年份（若有） < 父母死亡年份（若有）。

### 10. 为什么要做年份约束？

族谱数据必须符合基本现实逻辑。年份约束可以避免出现孩子早于父母出生、婚姻晚于孩子出生、死亡后结婚等错误。

### 11. 父母同一时间只能和一个人结婚如何实现？

创建或修改婚姻关系时检查已有婚姻区间。如果时间段重叠，则拒绝新增或修改，保证同一时间只有一个有效配偶。

### 12. 如何判断婚姻时间段重叠？

将未离婚视为开放区间，比较两个婚姻区间是否存在交集。如果存在交集，说明同一时间有多个婚姻关系。

### 13. 为什么允许离婚？

现实族谱中婚姻可能结束。系统保留离婚年份，既能表达历史婚姻，又不影响已经存在的亲子关系。

### 14. 为什么不能和自己结婚？

这是基本业务约束。婚姻双方必须是两个不同成员，后端会检查配偶 ID 不能相同。

### 15. 为什么婚姻双方必须在同一族谱？

当前系统以族谱为数据边界。跨族谱婚姻会让树结构、权限和导出范围复杂化，因此当前版本限制同族谱婚姻。

### 16. 查询祖先如何实现？

从当前成员开始，沿 `father_id` 和 `mother_id` 向上递归或队列遍历，记录距离当前成员的代数。

### 17. 查询四代曾孙如何实现？

从当前成员的子女开始，向下查找三层后代，得到第四代后代，再去重和排序。

### 18. 查询配偶和子女如何实现？

配偶来自婚姻表和共同子女推导；子女通过 `father_id` 或 `mother_id` 等于当前成员 ID 查询。

### 19. 为什么配偶查询既看婚姻表又看子女？

因为历史数据中可能存在亲子关系但没有显式婚姻记录。系统会通过共同子女补全配偶关系，增强容错。

### 20. 为什么查询姓名时要处理重名？

族谱中重名非常常见，单靠姓名不能唯一定位成员。系统在重名时提示候选成员和 ID，要求用户用 ID 确认。

### 21. `张_1_1` 查询为什么祖先为空是正常的？

`张_1_1` 是第一代成员，本身没有父母记录，所以祖先查询返回空列表是正确结果，不是用户不存在。

### 22. 为什么成员搜索支持模糊搜索，统计查询更偏精确匹配？

成员搜索用于浏览和发现对象，适合模糊搜索；统计查询会返回具体亲属关系，必须准确定位一个成员，所以使用精确姓名加重名 ID 确认。

### 23. 删除成员时如何处理关联婚姻？

删除成员会清理与该成员相关的婚姻关系，并更新或移除相关亲属关系，避免婚姻表保留无效成员引用。

### 24. 删除族谱时如何处理成员和婚姻？

删除族谱属于高影响操作，会删除该族谱下成员、婚姻和协作权限，保持数据库不残留孤立业务数据。

### 25. 协作权限如何判断？

如果用户是管理员、族谱创建者或被写入 `collaborations` 表，则可以编辑对应族谱；只有创建者可以授权或撤销协作者。

### 26. 为什么创建者也写入 collaborations？

这样权限判断可以统一处理“可编辑用户”。同时仍通过 `genealogies.creator_id` 判断谁是创建者，区分编辑权限和所有者权限。

### 27. 性别比统计如何实现？

按全库或指定族谱筛选成员，再统计 `gender` 字段中 `M`、`F`、`U` 的数量，前端用扇形图展示比例。

### 28. 长寿统计如何实现？

对有出生年份的成员计算寿命。若死亡年份为空，可按当前年份估算年龄。再按世代聚合平均寿命。

### 29. 早育查询如何实现？

先计算每个族谱每一代的平均出生年份，再筛选出生年份早于本代平均值的成员。

### 30. 超过 50 岁未婚男性查询如何实现？

筛选男性、未死亡、年龄大于 50 岁，并检查是否没有配偶或共同子女关系。

## 四、实际业务场景部分

### 1. 这个系统解决什么问题？

系统用于管理族谱数据，包括族谱创建、成员维护、亲属关系、婚姻关系、协作编辑、统计查询、导入导出和大规模数据性能验证。

### 2. 谁会使用这个系统？

主要用户包括族谱维护者、管理员、协作者，以及需要查询族谱信息的普通用户。

### 3. 管理员的业务职责是什么？

管理员负责用户管理、整库导入导出、测试数据生成导入，以及系统级数据维护。

### 4. 族谱创建者的业务职责是什么？

族谱创建者负责创建族谱、维护成员、授权协作者、撤销协作者和管理族谱内容。

### 5. 协作者的业务职责是什么？

协作者可以帮助维护指定族谱的成员信息、照片、父母关系和婚姻关系，但不能管理其他用户。

### 6. 普通用户注册后能立即做什么？

普通用户注册并登录后，可以创建自己的族谱。创建后默认成为该族谱的创建者，拥有编辑权限。

### 7. 为什么不需要手动设置族谱监理人？

创建族谱的人天然就是该族谱负责人。系统默认把当前登录用户设为创建者，减少用户填写成本和错误。

### 8. 为什么要支持协作权限？

现实族谱数据往往由多人共同维护。协作权限允许创建者邀请其他用户参与编辑，同时保留创建者的最终管理权。

### 9. 如果协作者误操作怎么办？

系统有用户操作日志，可以记录增删数据、授权、撤权、搜索等行为，便于追踪问题。高影响操作也有权限限制。

### 10. 为什么需要错误日志？

验收和实际运行中可能出现数据库连接失败、参数错误、代码异常等。错误日志可以帮助定位问题，不影响用户继续使用。

### 11. 为什么需要完整数据库导出？

完整导出用于备份、迁移和验收提交。通过 `fulldb/import_bundle.json` 可以快速恢复当前数据库状态。

### 12. 为什么需要导出单个成员信息？

实际使用中可能需要查看或提交某个成员的完整资料，包括本人、父母、子女、配偶、祖先和照片信息。

### 13. 为什么需要导入 CSV 新建族谱？

很多族谱数据可能来自 Excel 或其他系统。CSV 是通用格式，方便从外部数据源迁移到系统。

### 14. 为什么需要生成 10 万级数据？

课程要求验证数据库在较大数据规模下的查询和索引效果。10 万级数据可以更明显体现普通模式和索引模式的性能差异。

### 15. 为什么要生成 10 个族谱？

多个族谱可以测试族谱隔离、权限隔离、跨族谱查询、导入导出和全库统计。

### 16. 为什么其中一个族谱要有 6 万左右成员？

大族谱用于模拟真实大家族和压力场景，可以测试树形展示、成员搜索、索引性能和统计查询在大数据下的表现。

### 17. 命名为什么使用“姓氏_代_编号”？

这种命名格式便于验收时快速判断成员属于哪个姓氏、哪一代、编号多少，也方便搜索测试。

### 18. 为什么要支持模糊搜索？

实际用户往往只记得姓氏或部分姓名。模糊搜索可以快速筛选候选成员，再通过详情或 ID 精确定位。

### 19. 为什么统计查询中输入姓名还要支持 ID？

实际族谱中重名不可避免。姓名提升易用性，ID 保证最终操作对象准确。

### 20. 为什么要展示性别比？

性别比是族谱数据质量和人口结构的直观指标。它也能帮助发现数据异常，例如某个族谱大量未知性别。

### 21. 为什么要展示最长寿成员？

最长寿成员是族谱统计中的常见查询，能验证聚合查询和排序查询能力，也有实际展示价值。

### 22. 为什么要查询超过 50 岁未婚男性？

这是模拟业务统计条件查询，涉及性别、年龄、婚姻关系和亲子关系综合判断。

### 23. 为什么要查询早于平均生育年龄的成员？

这是模拟族谱中的人口学分析，可以观察不同世代的生育趋势和异常数据。

### 24. 为什么要查询四代曾孙？

四代曾孙查询可以验证多层后代追踪能力，也是族谱系统中典型的亲属范围查询。

### 25. 为什么要查询祖先？

祖先查询是族谱系统最核心的功能之一，可以从任意成员向上追溯父系和母系关系。

### 26. 为什么要把图片存数据库而不是本地？

验收和迁移时只需要数据库导出包即可恢复图片，不依赖本地文件夹。sha256 也方便去重和一致性检查。

### 27. 如果数据库为空，用户如何开始？

先执行 `init_db.sql` 初始化表结构，再注册或使用管理员账号创建族谱。也可以直接恢复 `fulldb/import_bundle.json`。

### 28. 如果用户 clone 项目后没有数据库怎么办？

按照 `README.md` 和 `instructions/env.md`：启动 TotemDB、创建 `genealogy` 数据库、执行 `init_db.sql`、恢复完整导出或导入生成数据。

### 29. 如果数据库名不叫 genealogy 怎么办？

可以通过环境变量 `TOTEM_DATABASE` 指定数据库名；启动脚本和导入导出脚本也读取这个环境变量。

### 30. 如果端口不是 55432 怎么办？

可以修改 `config/startup/startup.local.json` 或设置 `TOTEM_PORT` 环境变量。所有后端数据库访问都会读取该配置。

### 31. 如果页面提示 localhost 无法访问怎么办？

先确认 Uvicorn 是否运行，再查看 `/tmp/totem-genealogy-uvicorn.log`。如果 WSL 内能访问而 Windows 不能访问，可以用 WSL IP 访问。

### 32. 如果 TotemDB 提示已有 postmaster 怎么办？

先用 `ps -ef | grep '[t]master'` 检查是否已经运行。如果 `tsql` 能连接，就不需要重复启动。

### 33. 如果启动脚本失败怎么办？

查看脚本输出、数据库日志 `/tmp/totem-55432.log` 和 Web 日志 `/tmp/totem-genealogy-uvicorn.log`。也可以按 README 手动启动数据库和 Web。

### 34. 项目如何证明不是只做了页面？

项目包含真实 TotemDB 表结构、数据导入导出、10 万级数据生成、索引性能实验、EXPLAIN 输出和实际 SQL 查询逻辑，页面只是操作入口。

### 35. 项目还可以继续改进什么？

可以进一步加入数据库外键、分页优化、真正的 token/session 登录、批量编辑、审计日志入库、更强的数据校验和更完整的测试用例。

## 五、答辩快速总结

### 1. 项目核心亮点是什么？

核心亮点是把族谱业务、真实数据库、10 万级数据、索引性能实验、导入导出和可视化树结合在一起，不只是简单 CRUD。

### 2. 最核心的数据逻辑是什么？

最核心的是成员自关联父母关系、婚姻关系、亲子关系自动维护，以及围绕这些关系实现祖先、后代、配偶、子女和统计查询。

### 3. 最核心的数据库能力是什么？

包括表结构设计、批量数据导入、SQL 查询、索引性能对比、EXPLAIN 分析、完整数据库导出和恢复。

### 4. 最核心的业务能力是什么？

包括多用户管理、族谱创建、协作编辑、成员详情、亲属关系维护、照片存储、统计查询和数据迁移。

### 5. 如果只能演示三件事，建议演示什么？

建议演示：

1. 登录后查看首页性别比和族谱树。
2. 查询 `张_1_1` 的配偶/子女、祖先、四代曾孙。
3. 切换普通模式和性能模式搜索，展示耗时、内存和 EXPLAIN 输出。
